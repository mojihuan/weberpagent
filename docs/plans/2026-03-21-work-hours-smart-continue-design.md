# Work Hours 智能续接设计

## 问题背景

当前 work-hours skill 存在以下问题：

1. **工时内容与实际工作脱节**：工时文件通过"顺延"生成到 3-22，但实际 git commits 只记录到 3-15
2. **缺乏 commit 追踪**：没有记录"上次处理到哪个 commit"，导致无法续接新工作
3. **需要手动指定日期**：每次运行都要手动输入开始日期

## 解决方案

实现智能续接：自动检测上次处理位置，从断点继续记录新工作。

## 核心流程

```
启动 → 读取状态 → 扫描文件 → 显示预览 → 等待确认 → 生成工时 → 更新状态
```

## 详细设计

### 1. 状态文件 `.state.json`

```json
{
  "last_commit": "a7b4b8d",
  "last_file_date": "2026-03-22",
  "generated_files": ["2026-03-12.md", ...]
}
```

字段说明：
- `last_commit`: 上次处理到的最后一个 commit hash（短格式）
- `last_file_date`: 工时文件夹中最新的文件日期
- `generated_files`: 已生成的文件列表

### 2. 启动逻辑

**Step 1: 读取状态**
- 读取 `_backup/archives/工时/.state.json`
- 获取 `last_commit` 和 `last_file_date`

**Step 2: 扫描工时文件夹**
- 列出所有 `.md` 文件
- 找到最新的日期（文件名格式：`YYYY-MM-DD.md`）

**Step 3: 计算续接起点**
- `start_date = last_file_date + 1 天`

**Step 4: 获取新 commits**
```bash
git log <last_commit>..HEAD --format="%H %ad %s" --date=short --all
```

### 3. 状态预览

```
📋 工时续接状态
┌────────────────────┬─────────────────────────┐
│ 上次处理commit     │ a7b4b8d (2026-03-15)    │
│ 工时文件已到       │ 2026-03-22              │
│ 发现新commits      │ 32 个                   │
│ 续接起点           │ 2026-03-23              │
└────────────────────┴─────────────────────────┘

确认继续？ [Y/n]
```

### 4. 生成逻辑

- **模块分组**：按功能模块归类 commits
- **工时估算**：复杂度 × 1.2，向上取整到 0.5h
- **每日上限**：4 小时
- **溢出处理**：超过 4h 的部分顺延到下一天
- **时间段**：随机选择下午（14:00-18:00）或晚上（19:00-23:00）

### 5. 完成后更新

更新 `.state.json`：
- `last_commit` → 本次处理的最后一个 commit
- `last_file_date` → 本次生成的最新文件日期
- `generated_files` → 追加新生成的文件

## 边界情况

| 场景 | 处理方式 |
|------|----------|
| 无新 commits | 提示"无新工作需要记录"，退出 |
| .state.json 不存在 | 提示首次运行，需要手动指定起点 |
| 工时文件夹为空 | 从 last_commit 的日期 +1 开始 |
| 用户取消 | 不修改任何文件 |
