# 断言系统使用指南

## 概述

断言系统用于验证业务数据是否符合预期。本指南将帮助您理解如何配置和使用断言功能。

断言在 AI Agent 执行完测试步骤后自动运行，验证业务数据是否正确。每个断言包含：
- **断言方法**：指定要验证的业务场景
- **登录身份**：使用哪个账号获取数据
- **数据源**：从哪个数据源获取数据
- **三层参数**：API 查询条件、字段验证参数等

## 一、完整工作流程

### 1. 前端配置断言

1. 在任务编辑页面，找到"断言配置"区域
2. 选择断言类（PC 端使用 `PcAssert`）
3. 选择断言方法（如 `sell_sale_item_list_assert`）
4. 配置 Headers（登录身份），默认为 `main`
5. 配置 Data（数据源），默认为 `main`
6. 配置三层参数：
   - **API 参数**：设置查询条件
   - **字段参数**：选择要验证的字段和期望值

### 2. 执行测试

1. 保存任务后，点击"执行"按钮
2. AI Agent 将自动执行测试步骤
3. 测试步骤完成后，断言自动执行
4. 多个断言按配置顺序依次执行（非 fail-fast 模式）

### 3. 查看结果

1. 在报告详情页查看断言结果
2. 每个断言显示：
   - 通过/失败状态
   - 字段级验证结果（期望值 vs 实际值）
   - 执行耗时
   - 错误信息（如有）

## 二、三层参数详解

断言参数分为三层，每层有不同的用途：

| 参数层 | 用途 | 示例 | 说明 |
|--------|------|------|------|
| `api_params` | API 查询条件（过滤数据） | `salesOrder: 'SA'` | 控制从后端获取哪些数据 |
| `field_params` | 字段验证（期望值） | `saleTime: 'now-1m'` | 指定要验证的字段和期望值 |
| `params` | 向后兼容（旧格式） | 旧版参数 | 兼容旧配置，建议使用上面两层 |

### 第一层：API 参数 (api_params)

用于控制 API 查询条件，决定从后端获取哪些数据。

**常见参数：**

| 参数名 | 说明 | 示例值 |
|--------|------|--------|
| `salesOrder` | 销售单号前缀 | `'SA'` |
| `articlesStateStr` | 物品状态 | `'已销售'` |
| `i` | 订单状态 | `1`, `2`, `3` |
| `j` | 物品状态 | `13`, `3` |

**示例：**
```typescript
api_params: {
  salesOrder: 'SA',        // 销售单号以 SA 开头
  articlesStateStr: '已销售' // 物品状态为已销售
}
```

### 第二层：字段参数 (field_params)

用于指定要验证的字段和期望值。

**时间字段特殊处理：**

支持 "now" 表达式，系统在执行时自动转换为实际时间：

| 表达式 | 含义 | 示例输出 |
|--------|------|----------|
| `now` | 当前时间 | `2026-03-23 14:30:00` |
| `now-1m` | 1 分钟前 | `2026-03-23 14:29:00` |
| `now-5m` | 5 分钟前 | `2026-03-23 14:25:00` |
| `now-1h` | 1 小时前 | `2026-03-23 13:30:00` |
| `now-1d` | 1 天前 | `2026-03-22 14:30:00` |

**示例：**
```typescript
field_params: {
  saleTime: 'now-1m',       // 验证销售时间约为 1 分钟前
  articlesStateStr: '已销售' // 验证物品状态为已销售
}
```

### 第三层：通用参数 (params)

向后兼容层，旧版参数格式。建议优先使用 `api_params` 和 `field_params`。

如果同时提供了 `params` 和 `field_params`，系统会优先使用 `field_params`。

## 三、示例：销售出库断言

以下是一个完整的销售出库断言配置，用于验证销售出库操作是否成功：

```typescript
const assertionConfig = {
  className: 'PcAssert',
  methodName: 'sell_sale_item_list_assert',
  headers: 'main',           // 使用主账号登录
  data: 'main',              // 使用主数据源
  api_params: {
    salesOrder: 'SA',        // 销售单号前缀
    articlesStateStr: '已销售' // 物品状态
  },
  field_params: {
    saleTime: 'now-1m'       // 销售时间：约 1 分钟前
  }
}
```

**配置说明：**

| 字段 | 值 | 说明 |
|------|-----|------|
| `className` | `'PcAssert'` | PC 端断言类 |
| `methodName` | `'sell_sale_item_list_assert'` | 销售出库断言方法 |
| `headers` | `'main'` | 使用主账号登录（最常用） |
| `data` | `'main'` | 使用主数据源 |
| `api_params` | `{...}` | API 查询条件 |
| `field_params` | `{...}` | 字段验证参数 |

**可用的 Headers 选项：**

| 标识符 | 说明 |
|--------|------|
| `main` | 主账号（最常用） |
| `idle` | 闲置账号 |
| `vice` | 副账号 |
| `special` | 特殊账号 |
| `super` | 超级账号 |
| `platform` | 平台账号 |
| `camera` | 相机账号 |

**可用的 Data 选项：**

| 标识符 | 说明 |
|--------|------|
| `main` | 主数据 |
| `a` | 配件数据 |
| `b` | 分店数据 |
| `c` | 仓库数据 |
| `d` | 其他数据 |

## 四、报告解读

断言执行后，结果存储在报告的 `api_assertion_results` 字段中。

### 结果结构

```typescript
interface AssertionResult {
  success: boolean      // 执行是否成功（无异常）
  passed: boolean       // 断言是否通过
  fields: FieldResult[] // 字段级验证结果
  error: string | null  // 错误信息
  error_type: string | null  // 错误类型
  duration: number      // 执行耗时（秒）
}

interface FieldResult {
  name: string          // 字段名
  expected: string      // 期望值
  actual: string        // 实际值
  passed: boolean       // 是否通过
  comparison_type: 'equals' | 'contains'  // 比较类型
}
```

### 结果解读

| success | passed | 含义 |
|---------|--------|------|
| `true` | `true` | ✓ 断言通过 - 数据符合预期 |
| `true` | `false` | ✗ 断言失败 - 数据不符合预期，查看 fields 了解详情 |
| `false` | `false` | ⚠ 执行出错 - 如超时、参数错误，查看 error 了解详情 |

### 字段验证结果

当断言失败时，`fields` 数组包含每个字段的验证结果：

```typescript
fields: [
  {
    name: 'saleTime',
    expected: '2026-03-23 14:29:00',
    actual: '2026-03-23 10:00:00',
    passed: false,
    comparison_type: 'equals'
  },
  {
    name: 'articlesStateStr',
    expected: '已销售',
    actual: '已销售',
    passed: true,
    comparison_type: 'equals'
  }
]
```

## 五、开发者参考

### API 端点

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/external/assertions/methods` | GET | 获取可用断言方法列表 |
| `/api/external/assertions/fields` | GET | 获取可用字段列表 |
| `/api/tasks` | POST | 创建任务（包含断言配置） |

### 数据结构

**前端类型定义**（`frontend/src/types/index.ts`）：

```typescript
export interface AssertionConfig {
  className: string      // 断言类名，如 'PcAssert'
  methodName: string     // 断言方法名
  headers: string        // 登录身份标识符
  data: string           // 数据源标识符
  params: Record<string, number | string>      // 向后兼容参数
  api_params?: Record<string, string | number> // API 查询参数
  field_params?: Record<string, string>        // 字段验证参数
}
```

**后端执行函数**（`backend/core/external_precondition_bridge.py`）：

```python
async def execute_assertion_method(
    class_name: str,
    method_name: str,
    headers: str | None = 'main',
    data: str = 'main',
    api_params: dict | None = None,
    field_params: dict | None = None,
    params: dict | None = None,
    timeout: float = 30.0
) -> dict:
    """执行断言方法

    Args:
        class_name: 断言类名 ('PcAssert', 'MgAssert', 'McAssert')
        method_name: 断言方法名
        headers: 登录身份标识符
        data: 数据源标识符
        api_params: API 查询参数
        field_params: 字段验证参数
        params: 向后兼容参数
        timeout: 超时时间（秒）

    Returns:
        dict: 包含 success, passed, fields, error, duration
    """
```

## FAQ

### Q1: 断言失败但数据看起来正确？

**原因：** 时间参数使用了固定值而非 "now" 表达式。

**解决方案：** 使用时间预设如 `now-1m` 或 `now-5m`，让系统自动计算相对时间。

**示例：**
```typescript
// ❌ 错误：使用固定时间
field_params: { saleTime: '2026-03-23 14:30:00' }

// ✓ 正确：使用 now 表达式
field_params: { saleTime: 'now-1m' }
```

### Q2: 找不到要验证的字段？

**原因：** 字段列表来自 `base_assertions_field.py`，可能未包含该字段。

**解决方案：**
1. 在前端字段选择器中搜索字段名
2. 如果找不到，联系开发人员添加字段定义
3. 字段名区分大小写，确保拼写正确

### Q3: 断言执行超时？

**原因：** 默认超时时间为 30 秒，某些复杂查询可能超时。

**解决方案：**
1. 检查 API 参数是否过于宽泛导致数据量过大
2. 添加更精确的查询条件缩小数据范围
3. 联系开发人员调整超时配置

### Q4: Headers 选择错误导致数据查询失败？

**原因：** 不同的 headers 代表不同的登录账号，权限和数据范围不同。

**解决方案：** 根据测试场景选择正确的身份：

| 场景 | 推荐 Headers |
|------|--------------|
| 常规操作 | `main` |
| 分店相关 | `vice` |
| 平台管理 | `platform` |

### Q5: 字段参数未填写期望值？

**原因：** 选中字段但未填写期望值，断言无法执行验证。

**解决方案：**
1. 在前端配置界面，选中字段后必须填写期望值
2. 期望值不能为空字符串
3. 时间字段建议使用 `now` 表达式

### Q6: "now" 显示为字符串而非时间？

**原因：** 系统应在执行时将 "now" 转换为实际时间格式。

**解决方案：**
1. 确保使用最新版本的后端代码
2. 检查 `execute_assertion_method` 函数是否正确调用 `_convert_now_values`
3. 查看后端日志确认时间转换是否执行

### Q7: 如何查看断言执行的详细日志？

**解决方案：** 查看后端日志，搜索以下关键字：
- `Executing assertion` - 断言执行开始
- `Assertion.*passed` - 断言通过
- `Assertion.*failed` - 断言失败
- `Calling.*with kwargs` - 查看实际传入的参数

### Q8: 多个断言的执行顺序？

**说明：**
- 断言按配置顺序依次执行
- 非 fail-fast 模式：一个断言失败不影响其他断言执行
- 所有断言结果都会被收集到报告中

**最佳实践：**
1. 将关键断言放在前面
2. 避免断言之间有依赖关系
3. 每个断言验证独立的业务逻辑
