# AI + Playwright UI 自动化测试技术方案

## 1. 项目背景

### 1.1 背景说明

当前 Web UI 自动化测试主要依赖 Selenium、Playwright、Appium 等自动化框架。这类框架在浏览器驱动、页面操作、断言校验和测试集成方面已较为成熟，能够覆盖大量常规测试场景。然而，在实际业务落地过程中，传统 UI 自动化脚本仍然存在明显局限：

1. **对页面结构依赖强**
   自动化脚本通常依赖 DOM、XPath、CSS Selector、元素 ID、class 等结构信息进行元素定位。当前端页面发生改版、组件重构、文案调整、样式切换或动态渲染策略变化时，脚本容易失效。

2. **脚本维护成本高**
   在快速迭代项目中，前端页面变更频繁，导致自动化用例需要不断调整定位器、等待逻辑和断言方式，测试维护成本高，收益下降。

3. **复杂页面状态难以稳定覆盖**
   动态弹窗、懒加载、异步渲染、权限差异、网络延迟、区域滚动等复杂行为经常导致自动化用例出现偶发失败。

4. **视觉问题检测能力弱**
   传统自动化测试对文本断言、URL 断言、属性断言较强，但对布局错乱、元素遮挡、文字溢出、颜色异常、无障碍问题等“视觉层问题”覆盖有限。

5. **测试意图与执行细节脱节**
   业务上通常关注“登录系统”“修改配置”“提交审批”等业务目标，但自动化脚本层面却充满 `div > span > nth-child` 之类的底层定位细节，导致脚本可读性差、可迁移性弱。

随着多模态大模型（LLM / VLM）在图像理解、自然语言推理、页面语义识别和任务规划方面能力增强，将 AI 引入 UI 自动化测试已具备现实可行性。AI 可以用于页面理解、步骤规划、元素语义识别、异常反思、自愈修复和视觉校验，从而与 Playwright 的稳定执行能力结合，形成一套更智能、更鲁棒的 UI 自动化体系。

### 1.2 建设目标

本项目拟建设一套基于 **AI + Playwright** 的 UI 自动化测试能力，目标如下：

- 降低 UI 自动化脚本编写和维护成本
- 提升在复杂页面与动态场景下的自动化成功率
- 支持自然语言描述测试目标与测试步骤
- 增强元素定位失败后的自愈和异常恢复能力
- 将视觉理解能力引入 UI 测试过程
- 建立可追踪、可审计、可复盘的智能测试闭环
- 为后续统一 AI Testing Agent 平台建设打基础

## 2. 项目目标与范围

### 2.1 总体目标

建设一套面向 Web UI 场景的智能自动化测试能力平台，使测试系统具备以下能力：

- 能够理解页面截图与简化后的页面结构
- 能够根据自然语言任务描述判断当前页面状态和下一步动作
- 能够输出结构化测试动作并交由 Playwright 执行
- 能够在操作失败后基于新截图、错误信息和页面结构进行反思与重试
- 能够沉淀测试过程日志、截图、推理结果和执行报告

### 2.2 实施范围

#### 纳入范围
本期方案重点覆盖以下能力：

- Web 页面自动化测试
- 登录、搜索、表单填写、提交流程、配置修改等标准业务流程
- 元素定位失败后的 AI 自愈修复
- 基于截图的基础视觉验证
- 执行日志、失败截图、Trace 信息和报告归档
- 与现有 Playwright 测试体系兼容集成

#### 暂不纳入范围
以下内容不作为本期重点建设范围：

- 原生移动端复杂手势测试
- 游戏、视频流、3D 渲染等高实时 UI 场景
- 完全替代手工测试流程
- 无人值守执行高风险生产操作
- 跨桌面客户端和远程桌面纯视觉自动化

## 3. 现状问题分析

### 3.1 传统 UI 自动化面临的问题

#### 3.1.1 定位方式脆弱
当前多数自动化脚本依赖以下方式定位元素：

- ID
- CSS Selector
- XPath
- class
- DOM 层级路径

此类定位方式在页面结构稳定时效果较好，但在以下场景中容易失效：

- 前端组件重构
- 动态 class 生成
- 文案变化
- 低代码平台自动渲染
- 多端适配引起的层级变化

#### 3.1.2 脚本维护成本高
当 UI 页面迭代频繁时，用例维护通常体现在：

- 重新录制或修改选择器
- 调整等待时机
- 修改断言内容
- 兼容弹窗、加载状态和动画干扰

久而久之，自动化测试从“节约人力”变成“持续维修脚本”。

#### 3.1.3 动态页面带来的不稳定性
常见不稳定来源包括：

- 首次加载较慢
- 接口响应不稳定
- 验证码或弹窗干扰
- 多 tab 切换
- 权限差异
- 元素虽在 DOM 中但不可点击

传统脚本一旦失败，多数情况直接终止，需要人工排查。

#### 3.1.4 缺乏视觉层理解
UI 问题并不总是结构错误，常见视觉问题包括：

- 表单错位
- 按钮被遮挡
- 标题换行截断
- 颜色不符合规范
- 页面文字与实际设计不一致

传统自动化主要擅长结构性断言，对上述问题识别有限。

### 3.2 引入 AI 的价值

引入 AI 后，系统不再只依赖底层 DOM 规则，而是增加“页面理解”和“任务规划”能力，带来的价值主要体现在：

- 测试步骤可用自然语言描述
- 元素定位可以结合视觉与语义进行
- 执行失败后能够进行自愈修复
- 能够感知截图中的视觉问题
- 更适合处理动态变化页面
- 更利于构建自然语言驱动的智能测试系统

## 4. 技术方案概述

### 4.1 方案核心思想

本方案采用 **“AI 决策 + Playwright 执行”** 的混合架构。

其中：

- **AI 负责理解与规划**
  - 理解截图
  - 理解简化 DOM / Accessibility Tree
  - 根据任务目标决定下一步操作
  - 分析错误并生成修复策略

- **Playwright 负责浏览器控制与稳定执行**
  - 打开页面
  - 查找元素
  - 点击、输入、等待、截图
  - 记录执行日志与 Trace

该模式避免了纯视觉坐标点击在稳定性上的不足，同时利用 AI 提升对复杂页面和异常场景的处理能力。

### 4.2 方案核心闭环

系统整体工作闭环如下：

1. Playwright 打开目标页面
2. 获取当前页面截图
3. 提取简化 DOM / Accessibility Tree
4. 将截图、结构化页面信息、任务说明发送给 AI
5. AI 输出下一步结构化动作
6. Playwright 解析动作并执行
7. 再次观察页面变化
8. 若失败，则带着错误上下文进入 AI 反思与重试
9. 循环直到任务完成或达到上限

该闭环实现了“看—想—做—查—改”的智能执行流程。

## 5. 总体架构设计

### 5.1 架构分层

系统建议分为五层：

#### 5.1.1 感知层
负责采集和整理当前页面状态，包括：

- 页面截图
- Accessibility Tree 抽取
- DOM 精简与结构化
- OCR 文本识别（可选）
- 页面元信息提取（URL、Title、可见弹窗等）

#### 5.1.2 决策层
负责调用大模型完成智能推理，包括：

- 当前页面阶段识别
- 任务分解
- 下一步动作规划
- 执行结果判断
- 错误反思与修复策略生成

#### 5.1.3 执行层
由 Playwright 承担，负责：

- 浏览器实例管理
- 页面操作执行
- 元素定位
- 自动等待
- 截图 / 视频 / Trace 采集
- 错误抛出与反馈

#### 5.1.4 编排层
负责整个 Agent 执行循环的编排控制，包括：

- 上下文管理
- 最大步数限制
- 重试与回退控制
- 会话状态维护
- 动作白名单与安全策略

#### 5.1.5 结果层
负责：

- 测试报告输出
- 失败归因展示
- 执行轨迹留档
- 统计指标沉淀
- 用例资产化管理

## 6. 核心流程设计

### 6.1 页面感知流程

每轮执行前，系统都应重新获取页面当前状态，确保 AI 决策基于最新页面信息进行。页面感知流程如下：

1. Playwright 截取当前页面截图
2. 获取可见区域内的重要元素信息
3. 提取 Accessibility Tree 或精简后的交互元素清单
4. 过滤隐藏、禁用、不可见和无交互意义的节点
5. 将结果整理为标准结构供 AI 使用

该流程的目的，是同时向 AI 提供：

- 页面“长什么样”
- 页面“有哪些可交互元素”
- 当前元素“处于什么状态”

### 6.2 任务规划流程

AI 在每一轮决策中，需要结合以下输入进行判断：

- 当前任务目标
- 当前截图
- 当前结构化页面信息
- 历史已执行步骤
- 上一步动作
- 上一步执行结果
- 错误信息（如果存在）

AI 输出的内容应包括：

- 当前状态判断
- 下一步是否继续
- 下一步建议执行的动作
- 如失败，建议的修复策略

### 6.3 操作执行流程

编排层收到 AI 输出后，应进行以下处理：

1. 校验动作结构是否合法
2. 判断动作是否在允许的白名单中
3. 将结构化动作转换为 Playwright API 调用
4. 交由执行层实际执行
5. 收集执行成功/失败结果
6. 记录动作耗时、页面变化和异常信息

### 6.4 反思与自愈流程

当执行失败时，系统不立即终止，而是进入自愈流程：

1. 捕获错误堆栈
2. 重新截图并提取最新页面结构
3. 将失败动作、错误内容、新页面状态发送给 AI
4. AI 分析失败原因并生成新策略
5. 若策略合理，则再次执行
6. 若达到上限或判断为高风险，则终止并输出失败报告

## 7. 简化 DOM / Accessibility Tree 方案

### 7.1 设计目的

原始 DOM 内容通常冗长、噪声多、与业务语义弱相关，不适合直接提供给大模型。引入“简化 DOM / Accessibility Tree”的主要目的包括：

- 降低 Token 消耗
- 去除无意义结构噪音
- 提升 AI 对页面交互语义的理解效率
- 为元素定位和语义匹配提供结构化依据

### 7.2 核心保留字段

建议抽取如下字段：

- 元素唯一标识
- role
- text / name
- aria-label
- placeholder
- visible
- enabled
- interactive
- bounding box（可选）
- 页面层级（可选）

### 7.3 示例结构

```json
[
  {
    "id": "elem_001",
    "role": "textbox",
    "name": "用户名",
    "visible": true,
    "enabled": true,
    "interactive": true
  },
  {
    "id": "elem_002",
    "role": "button",
    "name": "登录",
    "visible": true,
    "enabled": true,
    "interactive": true
  }
]
```

### 7.4 实现建议

建议优先采用以下方式：

- 使用 Playwright 的 Accessibility Snapshot 能力提取基础无障碍树
- 辅以自定义 JS 注入，对交互元素进行补充抽取
- 将不同来源统一整理为标准 JSON 结构

这样既能兼顾语义质量，也能兼顾工程灵活性。

## 8. AI 决策机制设计

### 8.1 输入上下文设计

AI 每轮决策所接收的上下文建议包括：

- 当前任务描述
- 当前截图
- 当前简化 DOM / A11y Tree
- 上一步动作
- 执行错误信息
- 执行历史摘要
- 当前 URL / 页面标题
- 特定业务上下文（如当前用户、当前模块）

### 8.2 输出格式设计

从安全性和可控性考虑，本方案不建议默认让 AI 直接返回任意代码，而建议优先返回结构化动作。例如：

```json
{
  "status": "continue",
  "reason": "当前位于登录页，应先输入用户名",
  "action": {
    "type": "fill",
    "target": {
      "by": "role",
      "value": "textbox",
      "name": "用户名"
    },
    "value": "admin"
  }
}
```

#### 可支持的状态字段
- `continue`：继续下一步
- `done`：任务完成
- `retry`：建议重试
- `abort`：建议终止

#### 可支持的动作类型
- goto
- click
- fill
- press
- select
- scroll
- wait
- screenshot
- assert（可选）

### 8.3 结构化输出的优势

采用结构化动作而非自由代码，优势在于：

- 降低任意代码执行风险
- 易于白名单校验
- 易于审计、留档、回放
- 便于统一执行引擎解析
- 更适合平台化与服务化

## 9. Playwright 执行引擎设计

### 9.1 职责定位

Playwright 执行引擎作为底层执行组件，主要承担：

- 浏览器和上下文管理
- 页面打开与切换
- 元素查找与交互
- 自动等待和稳定性处理
- 失败截图、视频、Trace 采集
- 返回可供 AI 再次分析的运行结果

### 9.2 动作支持范围

建议至少支持以下动作：

- 页面访问：`goto`
- 点击：`click`
- 输入：`fill`
- 键盘：`press`
- 下拉选择：`selectOption`
- 鼠标悬停：`hover`
- 滚动：`scroll`
- 等待：`waitFor`
- 截图：`screenshot`

### 9.3 定位策略优先级

为提高稳定性，建议执行层按如下优先级定位：

1. `getByRole`
2. `getByLabel`
3. `getByText`
4. `getByPlaceholder`
5. CSS Locator
6. XPath
7. 坐标点击（最后兜底）

优先使用语义定位，可使脚本更贴近用户视角，也更利于 AI 理解和修复。

## 10. 自愈能力设计

### 10.1 自愈触发条件

以下场景触发自愈流程：

- 元素不存在
- 元素不可点击
- 页面未跳转
- 文案变化导致断言失败
- 弹窗阻塞主流程
- 按钮被遮挡或页面未滚动至可视区

### 10.2 自愈策略

系统可支持以下自愈策略：

1. **替换定位方式**
   从 CSS / XPath 切换为 role / text / label 定位。

2. **重新感知页面**
   通过新截图和结构信息重新判断元素位置和语义。

3. **处理阻塞状态**
   自动关闭弹窗、提示框、遮罩层。

4. **滚动与重试**
   先滚动页面，再重新点击。

5. **修正执行顺序**
   例如本应先展开菜单再点击子项。

6. **回退并重新规划**
   当当前页面状态与预期偏差较大时，重新制定执行路径。

### 10.3 自愈边界控制

为避免错误操作扩大，需设置边界：

- 每个任务设置最大重试次数
- 高风险动作默认不自动重复执行
- 达到阈值后强制终止并人工介入
- 删除、支付、发布、提交类操作需加入额外确认机制

## 13. 安全与风险控制

### 13.1 风险识别

AI 引入后，主要风险包括：

- 模型幻觉导致定位错误
- 任意代码执行风险
- 敏感信息泄露
- 高风险页面误操作
- 模型输出不稳定导致同任务行为不一致

### 13.2 控制措施

针对上述风险，建议采用以下控制措施：

1. **结构化动作输出**
   不直接执行自由生成代码，优先使用白名单动作。

2. **动作白名单限制**
   仅允许系统定义好的操作集合。

3. **高风险动作人工确认**
   提交、删除、支付等操作增加确认或审批机制。

4. **数据脱敏**
   账号、密码、手机号、订单号等敏感信息在发送给模型前进行脱敏或替换。

5. **上下文范围限制**
   限定模型可见页面范围、可访问域名和最大步数。

6. **模型结果校验**
   对 AI 返回结果进行 schema 校验和合法性校验。

## 14. 技术选型建议

### 14.1 核心组件建议

- **浏览器自动化**：Playwright
- **多模态大模型**：支持图像理解和文本推理的视觉模型
- **OCR 能力**：PaddleOCR / Tesseract（按需启用）
- **执行编排**：LangChain / 自研 Orchestrator
- **日志存储**：PostgreSQL / Elasticsearch / 对象存储
- **缓存与状态**：Redis
- **报告展示**：Playwright HTML Report + 自定义测试看板

### 14.2 选型原则

- 稳定优先
- 可控优先
- 兼容现有自动化资产
- 支持私有化部署或受控调用
- 便于审计与成本管理

## 17. 预期收益

本方案落地后，预期收益包括：

1. **降低维护成本**
   当页面变更时，无需频繁手工修改底层定位器。

2. **提升自动化覆盖能力**
   对复杂页面、动态交互和视觉问题的覆盖能力增强。

3. **提升用例开发效率**
   测试步骤可更贴近业务语义，降低脚本编写门槛。

4. **提升问题定位效率**
   测试报告中保留了推理、截图、动作和错误上下文，便于快速定位问题。

5. **为智能测试平台打基础**
   为后续自然语言生成用例、自动缺陷归因、智能回归等能力预留扩展空间。

## 19. 结论

本方案以 Playwright 作为稳定执行底座，以 AI 作为页面理解、任务规划和自愈修复能力层，构建一套“AI 决策 + 自动化执行”的智能 UI 测试体系。该体系兼顾了传统自动化框架的稳定性与多模态大模型的理解能力，能够有效缓解当前 UI 自动化测试中脚本脆弱、维护成本高、视觉问题难覆盖和异常恢复差等问题。

从落地路径看，本方案不建议一开始追求“完全替代人工测试”或“完全 AI 自主测试”，而是建议从 **AI 辅助执行、自愈修复、关键场景试点** 切入，逐步演进为平台化、规模化的智能测试能力。
