### 2026-03-11: 基于区域的元素定位消歧

| 字段 | 内容 |
|------|------|
| **时间** | 2026-03-11 |
| **触发原因** | 页面上存在多个相同文本的元素（如侧边栏和主内容区都有"商品采购"），Playwright strict mode 报错：`strict mode violation: get_by_text("商品采购") resolved to 2 elements` |
| **调优经过** | 1. **类型层**：`InteractiveElement` 添加 `region` 字段<br>2. **感知层**：`_extract_elements` 添加区域检测（使用 `el.closest()` 检测 sidebar/header/main/footer/modal）<br>3. **执行层**：`_locate_element` 添加区域消歧逻辑，当 `count > 1` 时用区域选择器限定<br>4. **Prompt 层**：`format_elements_for_prompt` 展示区域信息，让 LLM 能看到上下文 |
| **调优结果** | ✅ 成功 - 所有单元测试通过，元素列表现在展示区域信息（如 `区域: 侧边栏`），执行层能智能消歧 |
| **下一步建议** | 1. 运行完整 Agent 测试验证实际效果<br>2. 考虑将 modal 区域检测优先级提到最高（弹窗元素优先处理）<br>3. 后续可添加下拉菜单/弹出层区域检测 |

**设计文档**：`docs/plans/2026-03-11-region-based-element-location-design.md`

**实现效果示例**：
```
[0] BUTTON | 文本: "商品采购" | 区域: 侧边栏
[1] DIV | 文本: "商品采购" | 区域: 主内容区
```

执行层自动处理：
```python
# 原来: page.get_by_text("商品采购") → 报错 strict mode violation
# 现在: page.locator('aside, .sidebar, nav').get_by_text("商品采购") → 精确定位侧边栏元素
```

### 2026-03-11: 循环检测和恢复机制增强

| 字段 | 内容 |
|------|------|
| **时间** | 2026-03-11 11:30 |
| **触发原因** | 从日志分析，Agent 在 Step 7-21 连续执行 `wait` 动作 15 次，循环检测被触发但恢复机制无效（只是简单的 wait/scroll，没有改变 LLM 的决策）。另外，当弹窗出现后，点击"确定"按钮因多个匹配元素触发 Playwright strict mode 违规，反思机制尝试用"确 定"（带空格）仍然失败。 |
| **调优经过** | 1. **执行层 - 元素定位**：`_locate_element` 方法在区域限定后仍有多个匹配时，直接返回 `.first`，避免 strict mode 违规<br>2. **执行层 - 点击动作**：`_click` 方法增加策略 1.5，当 `strict mode violation` 时尝试 `.first` 点击<br>3. **Agent 层 - 循环检测**：`_detect_loop()` 返回值改为 `(is_loop, loop_type)`，提供循环类型描述<br>4. **Agent 层 - 恢复策略**：`_recover_from_loop(loop_type)` 根据循环类型智能选择恢复策略（相同动作→滚动，高失败→滚动查找）<br>5. **Agent 层 - 上下文增强**：`_build_history_context()` 添加循环警告信息，让 LLM 知道它陷入了循环，需要换策略 |
| **调优结果** | ⏳ 待验证 - 代码已完成，需运行完整测试验证 |
| **下一步建议** | 1. 运行采购单场景测试验证效果<br>2. 考虑在 SYSTEM_PROMPT 中增加循环避免的指导规则<br>3. 如果仍然无效，考虑增加"强制跳过"机制：连续 N 次相同动作后自动跳过 |

**详细记录**：

#### 修改文件
- `executor.py`:
  - `_locate_element()`: 区域限定后仍有多个匹配时返回 `.first`
  - `_click()`: 增加 strict mode 违规时的 `.first` 后备策略

- `agent.py`:
  - `_detect_loop()`: 返回 `(is_loop, loop_type)` 元组
  - `_recover_from_loop(loop_type)`: 根据循环类型智能选择恢复策略
  - `_build_history_context()`: 添加循环警告信息
  - `__init__`: 新增 `_last_loop_type` 属性
  - `run()`: 更新循环检测调用，记录 `_last_loop_type`

#### 循环警告输出示例
```
⚠️ 循环警告：检测到 连续执行相同动作: wait -> None
请换一种策略，不要重复之前的动作！

Step 7: wait ✅
Step 8: wait ✅
Step 9: wait ✅
```

#### 问题根因分析

1. **循环检测触发但恢复无效**：
   - 原因：恢复只是 `wait`，但问题不是等待，而是 LLM 一直重复相同的决策
   - 解决：给 LLM 提供明确的循环警告上下文，让它知道需要换策略

2. **Strict mode 违规**：
   - 原因：页面上有多个"确定"按钮（弹窗内可能也有多个）
   - 解决：在区域限定后仍有多个匹配时，使用 `.first` 而不是报错

## 调优经验总结

### 常见问题与解决方案

| 问题 | 表现 | 解决方案 |
|------|------|----------|
| LLM 输出数字索引 | `target: "2"` 而非元素文本 | Prompt 强制规则 + Few-shot 示例 + 反思修正 |
| 元素定位失败 | 找不到元素 | 多策略定位（8 种）+ JavaScript 回退 |
| 侧边栏导航循环 | 重复点击菜单项无变化 | hover 优先 + 菜单检测 + 等待验证 |
| 任务完成判断不准 | 提前 done 或超时 | 完成证据规则 + Few-shot 示例 |

### 优化优先级

1. **Prompt 层**（成本最低，效果最直接）
2. **执行层**（提高鲁棒性）
3. **感知层**（提供更好的信息）
4. **反思层**（处理异常情况）
