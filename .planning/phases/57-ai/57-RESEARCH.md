# Phase 57: AI 推理格式优化 - Research

**Researched:** 2026-04-02
**Domain:** Frontend text parsing + visual formatting (React + Tailwind CSS)
**Confidence:** HIGH

## Summary

This phase is a pure frontend formatting change. The backend produces reasoning text as a pipe-delimited string (`" | ".join(parts)`) with three top-level labels: `Eval:`, `Memory:`, `Goal:`. The LLM also embeds a `Verdict:` sub-label inside the Eval value. The task is to parse this string on the frontend, split it into labeled segments, and render each segment on its own line with a colored badge.

The scope is narrow: two components (`ReasoningLog.tsx` and `StepItem.tsx`) each need their reasoning rendering replaced with a shared parsing + formatting utility. No backend changes, no database migration, no new dependencies.

**Primary recommendation:** Extract a `parseReasoning()` utility function and a `ReasoningText` display component. Both components import and use them. Keep the parser simple: split by ` | `, match labels case-insensitively, handle Verdict as a secondary split within the Eval segment.

### Key Data Observation: Verdict Is NOT a Top-Level Label

`Verdict:` appears **inside** the `Eval:` value text, generated by the LLM as part of its evaluation narrative. It is NOT a separate pipe-delimited segment. The pattern is:

```
Eval: [evaluation text] Verdict: [verdict value] | Memory: [memory text] | Goal: [goal text]
```

**Parsing strategy:**
1. Split by ` | ` to get top-level segments
2. Each segment starts with a label (`Eval:`, `Memory:`, `Goal:`)
3. For the `Eval:` segment, additionally extract `Verdict:` from the value portion
4. Render Verdict as a sub-badge within the Eval line (or as its own line — planner decides)

### Edge Cases Found in Data

| Case | Example | Handling |
|------|---------|----------|
| Missing segments | Only `Eval:` and `Goal:`, no `Memory:` | Parser handles missing segments gracefully |
| Verdict values vary | `N/A`, `Success`, `Partial success - navigation worked...`, `需要继续操作` | Verdict text is free-form, display as-is |
| All segments missing | `reasoning = ""` or `null` | Show "暂无推理记录" (existing behavior, unchanged) |
| Non-standard text | Text that does not match any label | Display as plain text line (D-07) |

### Pattern 1: Parse-Then-Render Separation

**What:** Separate the parsing logic from the rendering logic. The parser returns a structured array, the renderer maps over it.

**When to use:** Any text that needs structured display from a flat string.

**Example:**

```typescript
// utils/reasoningParser.ts
interface ReasoningSegment {
  label: string       // 'Eval' | 'Verdict' | 'Memory' | 'Goal'
  content: string     // The text value after the label
}

export function parseReasoning(text: string): ReasoningSegment[] {
  if (!text || !text.trim()) return []

  const segments: ReasoningSegment[] = []

  // Split by " | " delimiter
  const parts = text.split(' | ')

  for (const part of parts) {
    // Match top-level labels: Eval:, Memory:, Goal:
    const topMatch = part.match(/^(Eval|Memory|Goal)\s*:\s*(.*)/i)
    if (topMatch) {
      const label = topMatch[1]
      const value = topMatch[2].trim()

      if (label.toLowerCase() === 'eval') {
        // Check for embedded Verdict within Eval value
        const verdictMatch = value.match(/Verdict\s*:\s*(.*)/i)
        if (verdictMatch) {
          const evalContent = value.substring(0, value.indexOf(verdictMatch[0])).trim()
          const verdictContent = verdictMatch[1].trim()
          if (evalContent) {
            segments.push({ label: 'Eval', content: evalContent })
          }
          segments.push({ label: 'Verdict', content: verdictContent })
        } else {
          segments.push({ label: 'Eval', content: value })
        }
      } else {
        segments.push({ label, content: value })
      }
    } else if (part.trim()) {
      // Non-standard text — return as plain segment
      segments.push({ label: '', content: part.trim() })
    }
  }

  return segments
}
```

## Common Pitfalls

### Pitfall 1: Verdict Is Embedded in Eval, Not a Separate Segment
**What goes wrong:** Parsing `Verdict:` as a top-level pipe-delimited label when it is actually embedded inside the Eval value text.
**Why it happens:** The CONTEXT.md lists "Eval/Verdict/Memory/Goal" as if they are peers, but database data shows Verdict is always inside the Eval text.
**How to avoid:** Split by ` | ` first for top-level segments (Eval, Memory, Goal), then scan the Eval content for `Verdict:` as a secondary parse.
**Warning signs:** If Verdict content appears in Memory or Goal segments, the parser is wrong.

### Pitfall 2: Tailwind v4 Class Names
**What goes wrong:** Using `bg-purple-100 text-purple-700` without verifying the classes work in Tailwind v4.
**Why it happens:** Tailwind v4 changed from config-based to CSS-first configuration, but all standard utility classes still work the same way.
**How to avoid:** Tailwind v4.2.x supports all standard color utility classes (bg-{color}-{shade}, text-{color}-{shade}). The project already uses these patterns extensively. No issue expected.

### Pitfall 3: Verdict Text Contains Pipe Characters
**What goes wrong:** If the LLM generates a pipe character inside an Eval or Verdict text, splitting by ` | ` would incorrectly break the segment.
**Why it happens:** The backend delimiter is ` | ` with spaces, while natural text pipes are usually without spaces.
**How to avoid:** This is a low-probability edge case. The ` | ` delimiter (with spaces) is unlikely to appear in natural language text. If needed, split only on the first N occurrences matching the known label pattern.
**Warning signs:** Segments that don't start with a known label after splitting.

### Pitfall 4: Empty String vs Null Reasoning
**What goes wrong:** Treating empty string `""` and `null` differently when both should show "暂无推理记录".
**Why it happens:** The backend returns `""` when parts is empty, but null is also possible from the database.
**How to avoid:** Check `!text || !text.trim()` in the parser, and `!step.reasoning` at the component level. Both paths lead to the "暂无推理记录" fallback.

### ReasoningText Component Sketch

```tsx
// components/shared/ReasoningText.tsx
import { parseReasoning } from '../../utils/reasoningParser'

const BADGE_STYLES: Record<string, string> = {
  Eval: 'bg-purple-100 text-purple-700',
  Verdict: 'bg-green-100 text-green-700',
  Memory: 'bg-orange-100 text-orange-700',
  Goal: 'bg-blue-100 text-blue-700',
}

interface ReasoningTextProps {
  text: string
}

export function ReasoningText({ text }: ReasoningTextProps) {
  const segments = parseReasoning(text)

  if (segments.length === 0) {
    return <span className="text-sm text-gray-600">{text}</span>
  }

  return (
    <div className="space-y-1">
      {segments.map((segment, index) => (
        <div key={index} className="flex items-start gap-2">
          {segment.label ? (
            <>
              <span className={`inline-flex items-center px-2 py-0.5 rounded text-xs font-medium shrink-0 ${BADGE_STYLES[segment.label] ?? 'bg-gray-100 text-gray-600'}`}>
                {segment.label}
              </span>
              <span className="text-sm text-gray-600">{segment.content}</span>
            </>
          ) : (
            <span className="text-sm text-gray-600">{segment.content}</span>
          )}
        </div>
      ))}
    </div>
  )
}
```

## Open Questions

1. **Verdict rendering — sub-badge or separate line?**
   - What we know: Verdict is always embedded in the Eval value text. Sample values: `N/A`, `Success`, `Partial success...`, `需要继续操作`.
   - What's unclear: Should Verdict be rendered as a sub-badge on the same line as Eval content, or as its own separate line?
   - Recommendation: Render as its own line (as a separate badge row) since the Eval content and Verdict are conceptually distinct. This matches the user's mental model of "each item gets its own line."

2. **Should `shared/index.ts` be updated to export ReasoningText?**
   - What we know: The barrel export file exists at `frontend/src/components/shared/index.ts` and exports all other shared components.
   - Recommendation: Yes, add `export { ReasoningText } from './ReasoningText'` for consistency.
