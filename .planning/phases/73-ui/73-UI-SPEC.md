## Interaction Specification

### Polling Behavior
- Initial fetch on mount (no delay)
- Subsequent fetches every 2000ms via `setInterval`
- Stop polling when `batch.status === 'completed'`
- Clear interval on component unmount (useEffect cleanup)
- If fetch fails, continue polling (do not stop on error)
- Track completion with `useRef` to avoid stale closure issues

### Navigation
- On `batchesApi.create()` success in Tasks.tsx: `navigate('/batches/' + response.id)` immediately
- On BatchTaskCard click: `navigate('/runs/' + run.id)` for any run status
- Back navigation: user clicks sidebar "Tasks" to return to task list

### Toast Notification
- Trigger: when `batch.status` transitions to `completed` (detect by comparing previous and current batch status)
- Content: `批量执行完成：{success} 成功，{failed} 失败`
- Type: `toast.success()` if all tasks succeeded, `toast.warning()` if any failed
- Fire only once (use ref to track whether toast was already shown)

### Loading States
1. **Initial load**: Full-page `LoadingSpinner` centered in content area
2. **Runs empty (race condition)**: `LoadingSpinner` + text `正在加载任务列表...` in card area
3. **Runs populated, still polling**: Cards update in place, no loading overlay
4. **Error state**: Red error message + button labeled `刷新重试` in card area

## State Transitions

```
Page Load
  |
  v
[loading] --(first fetch success)--> [displaying cards, polling]
  |                                        |
  |                                   (batch.status === 'completed')
  |                                        |
  v                                        v
[error] <--(fetch fails)------------- [complete, toast shown, stopped]
  |
  v
("刷新重试" button) --> [loading] (re-fetch)
```
