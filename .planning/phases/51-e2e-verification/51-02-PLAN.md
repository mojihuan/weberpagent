    **VAL-02 Check: No element repeated > 2 consecutive failures (per D-05)**

    ```bash
    uv run python -c "
    import json
    from collections import defaultdict

    # Track consecutive failures per target_index
    actions = []
    with open('outputs/${RUN_ID}/logs/run.jsonl') as f:
        for line in f:
            e = json.loads(line)
            if e.get('category') == 'agent':
                actions.append(e)

    # Build sequence of (action_name, target_index, evaluation) per step
    steps = []
    for a in actions:
        an = a.get('action_name', '')
        idx = a.get('action_params', {}).get('index')
        ev = a.get('evaluation', '') or ''
        steps.append((an, idx, ev))

    # Check consecutive failures per index
    failure_keywords = ['fail', 'error', 'not found', 'failed', 'unable', 'timeout', '没有', '失败', '无法']
    consecutive = defaultdict(int)
    violations = []
    for an, idx, ev in steps:
        if idx is None:
            continue
        is_failure = any(kw in ev.lower() for kw in failure_keywords)
        if is_failure:
            consecutive[idx] += 1
            if consecutive[idx] > 2:
                violations.append(f'Index {idx}: {consecutive[idx]} consecutive failures (step: {an})')
        else:
            consecutive[idx] = 0

    if violations:
        print('VAL-02 FAIL: Element repeated failure > 2 times:')
        for v in violations:
            print(f'  - {v}')
    else:
        print('VAL-02 PASS: No element repeated failure > 2 consecutive times')
    print(f'Total steps analyzed: {len(steps)}')
    "
    ```

    PASS if no violations found. FAIL if any index has > 2 consecutive failures.

    **VAL-04 Check: PreSubmitGuard structural evidence (per D-07)**

    IMPORTANT CAVEAT (from 51-RESEARCH.md Pitfall 1): PreSubmitGuard.check() is wired in _execute_actions() but currently called with actual_values=None and submit_button_text=None, so it returns should_block=False. This means:
    - The guard IS structurally integrated (call exists in code)
    - Active blocking requires DOM value extraction which is not yet implemented
    - A "Submit blocked" entry would only appear if the guard had real values to check

    Check for ANY evidence of PreSubmitGuard being reached:
    ```bash
    uv run python -c "
    import json
    guard_entries = []
    submit_blocked = []
    with open('outputs/${RUN_ID}/logs/run.jsonl') as f:
        for line in f:
            e = json.loads(line)
            msg = e.get('message', '')
            cat = e.get('category', '')
            if 'submit' in msg.lower() or 'guard' in msg.lower() or 'blocked' in msg.lower():
                guard_entries.append(e)
            if cat == 'monitor' and 'blocked' in msg.lower():
                submit_blocked.append(e)

    if submit_blocked:
        print('VAL-04 PASS: PreSubmitGuard actively blocked submit:')
        for e in submit_blocked:
            print(json.dumps(e, ensure_ascii=False, indent=2))
    elif guard_entries:
        print('VAL-04 PARTIAL: PreSubmitGuard-related entries found (structural evidence):')
        for e in guard_entries:
            print(json.dumps(e, ensure_ascii=False, indent=2))
    else:
        print('VAL-04 NOTE: No PreSubmitGuard entries found in log.')
        print('This is expected because PreSubmitGuard.check() is called with actual_values=None.')
        print('The guard IS structurally integrated (verified by unit test test_blocks_submit_click).')
        print('Active blocking requires DOM value extraction (not yet implemented).')
    "
    ```
