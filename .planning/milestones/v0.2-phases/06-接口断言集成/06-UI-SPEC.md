## Visual Specifications

### ApiAssertionInput (in TaskForm)

```
Layout:
  - Label: "接口断言 (可选)"
  - Description: "输入 Python 代码进行 API 断言，支持时间断言、数据匹配等"
  - Textarea: 4 rows, font-mono text-sm
  - Actions: [+ 添加接口断言] link, [删除] button per item

Styling:
  - Same as preconditions textarea
  - border-gray-200, rounded-lg
  - focus:ring-2 focus:ring-blue-500
  - font-mono text-sm for code
```

### ApiAssertionResults (in Report)

```
Layout:
  - Header: "接口断言结果" + pass rate badge
  - Results list: card per assertion
  - Each card: status icon + assertion name + field results

Styling:
  - pass: bg-green-50 border-green-200, CheckCircle icon green-500
  - fail: bg-red-50 border-red-200, XCircle icon red-500
  - Field result: text-sm, field name bold, expected/actual comparison
```

### FieldAssertionItem

```
Layout:
  - Icon: CheckCircle or XCircle
  - Field name (bold)
  - Assertion type badge (optional)
  - Error message if failed

Styling:
  - text-sm text-gray-700
  - pass: text-green-600
  - fail: text-red-600 with expected/actual values
```

## Notes

1. **No shadcn initialization recommended:** Project uses vanilla Tailwind CSS with established patterns. Introducing shadcn would add unnecessary complexity for this phase's scope.

2. **Reuse existing patterns:** ApiAssertionInput should mirror the preconditions input pattern exactly for consistency.

3. **Separate component for API assertions:** While existing AssertionResults could be extended, creating a separate ApiAssertionResults component provides clearer separation and allows for different display requirements (e.g., field-level results, assertion type badges).

4. **Decimal tolerance default:** Use 0.01 (1%) as the default tolerance for decimal approximation assertions, matching RESEARCH.md recommendation.
