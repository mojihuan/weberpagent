**For run_pipeline.py (576 lines), verify these specific concerns from RESEARCH.md:**

1. **Pipeline error propagation (Pitfall 1):** Trace the 6 stages of `run_agent_background()`:
   - Stage 1: `_run_preconditions()` -- Does it return None on failure? Does it publish `None` to event_manager before returning? Does the caller's finally block also publish None? (Double None issue from Pitfall 2 in RESEARCH.md)
   - Stage 2: `_run_auth_and_session()` -- Does exception handling fall back to text login correctly?
   - Stage 3: `agent_service.run_with_cleanup()` -- Does it raise on agent error? Is the error caught properly?
   - Stage 4-5: `_run_ui_assertions()` and `_run_external_assertions()` -- Do they return status strings instead of raising? Can they mask failures?
   - Stage 6: `_finalize_run()` and `_run_code_generation()` -- Does finalize always run? Is codegen wrapped in non_blocking_execute?

2. **Context mutation (Pitfall 3):** Find where `_run_external_assertions()` modifies the `context` dict (adds `external_assertion_summary`). Check if this key leaks into `variable_map` construction. The filter at line ~546 excludes keys starting with "assertion" but "external_assertion_summary" starts with "external_" not "assertion".

3. **Mutable dict closures:** Identify all mutable dicts used as closures (`counters`, `step_stats_data`, `_prev_dom_hash_data`, `agent_ref`) and verify they are correctly initialized and not shared across runs.

4. **Error state consistency:** If preconditions fail but preconditions publish a "finished" event, does the pipeline leave the run status in a consistent state? Check for paths where `run.status` could be left as "running" after an error.

**For agent_service.py (658 lines), verify these specific concerns:**

1. **Dual step callback (Pitfall 1):** Trace the callback wiring:
   - `self._create_step_callback()` creates a wrapper callback
   - This is passed as `register_new_step_callback` to MonitoredAgent
   - MonitoredAgent also creates its own internal `step_callback` via `create_step_callback()`
   - Verify the execution order: browser-use -> MonitoredAgent._step_callback -> agent_service callback -> on_step
   - Check if detectors only see action[0] while code buffer sees all actions

2. **Screenshot I/O in async context (Pitfall 4):** Find `save_screenshot()` and verify it calls `filepath.write_bytes()` which is synchronous blocking I/O in the async event loop. Check screenshot size handling.

3. **DOM serialization blocking (from CONCERNS.md):** Find `_extract_browser_state()` and verify it calls `dom_state.llm_representation()` + `hashlib.sha256()` synchronously on every step.

4. **Browser session lifecycle:** Verify the browser session is correctly created, cookie-injected, and cleaned up. Check for paths where the browser session could leak (not closed).

**For code_generator.py (553 lines):**

1. **Code assembly correctness:** Verify the generated Playwright test file structure:

   - Assertion injection: Are assertions correctly placed at the end?

4. **F-string injection risks:** The generator uses f-strings to embed values into Python code. Verify no values contain quotes or backslashes that could break the generated code.

**For step_code_buffer.py (414 lines):**

1. **Duplicate wait logic (Pitfall 6):** Verify that `_derive_wait()` adds wait code BEFORE actions and `assemble()` adds additional stability waits AFTER click actions. For click actions specifically:
   - `_derive_wait()` should produce a 3000ms wait BEFORE the click
   - `assemble()` should produce networkidle + 500ms wait AFTER the click
   - Is the 3000ms pre-click wait excessive? Does it match the comment "click default 3s for async updates"?

2. **Evaluate-to-fill conversion:** Trace how the buffer detects "evaluate" actions that are actually "fill" operations (corrective detection). Verify the conversion logic handles:
   - Single action steps
   - Multi-action steps (agent returns >1 action)
   - Edge cases: evaluate without value assignment, evaluate on non-input elements

3. **assemble() output correctness:** Verify the assembled code has:
   - Correct indentation for each step
   - Proper locator formatting
   - No duplicate steps
   - Wait strategies applied correctly

**For monitored_agent.py (227 lines):**

1. **PreSubmitGuard dead code (Pitfall 7):** Verify that `actual_values=None` and `submit_button_text=None` are hardcoded in the PreSubmitGuard.check() call. The guard logic always returns `should_block=False` because both are None. Document this as dead logic.

2. **Intervention injection:** Trace how `_pending_interventions` is populated in the step callback and consumed in `_prepare_context`:
   - Is the list cleared at the right time?
   - Can interventions accumulate if step_callback raises between append and clear?
   - Are detector failures properly caught (wrapped in try/except)?

3. **StallDetector integration:** Verify that stall detection result is correctly converted to an intervention message and injected into the agent's context.

4. **_execute_actions override:** Verify that the submit-blocking logic correctly prevents premature submit clicks and allows them when appropriate.
