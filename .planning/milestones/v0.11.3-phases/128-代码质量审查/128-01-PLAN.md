    ## Quick-scan methodology
    For each file in the codebase, perform a rapid review focusing on MAINT/ARCH-03/PERF-01 dimensions. Use grep/read to scan patterns, not line-by-line analysis.

    ### MAINT-01 (DRY/SOLID violations) -- scan for:
    1. **Code duplication**: Search for repeated patterns across files
       - grep for common patterns: `try:\n.*except Exception`, `async with async_session`, `json.dumps.*ensure_ascii`, `logging.getLogger`
       - Identify 3+ files with identical error handling boilerplate
       - Find repeated state management patterns in frontend hooks (useState+useEffect+fetch)
    2. **SRP violations**: Files handling >3 distinct concerns
       - run_pipeline.py (already flagged in Phase 125 as god-module, ref with "See 125-FINDINGS.md")
       - Any file >500 lines likely violates SRP
    3. **DRY violations in frontend**: All 4 hooks (useTasks, useReports, useDashboard, useBatchProgress) use identical manual fetch pattern instead of React Query (already found in Phase 127, ref with "See 127-FINDINGS.md")

    ### MAINT-02 (structural complexity) -- use radon/ESLint output from Task 1:
    - Files with grade C or worse in radon output
    - Functions exceeding 50 lines (grep for long function defs)
    - Files exceeding 500 lines

    ### MAINT-03 (misleading names, per D-04) -- scan for:
    1. Functions where name != behavior:
       - `check_element_exists` that never checks elements (See 125-FINDINGS.md)
       - `_execute_actions` in MonitoredAgent that also runs detectors
    2. Inconsistent abstraction level naming:
       - Mix of `get_xxx` and `fetch_xxx` and `load_xxx` for same operation type
       - Functions named `validate` that do transformation instead
    3. Unclear abbreviations: grep for single-letter variables (like `l` in action_translator.py, already noted in Phase 125, ref)

    ### ARCH-03 (cross-cutting consistency) -- scan three dimensions per D-08:
    1. **Error handling strategy consistency**:
       - Count files using `non_blocking_execute` vs raw `try/except` vs bare operations
       - Count files using `HTTPException` vs `error_response` vs `raise_not_found`
       - Check if `error_utils.py` helpers are used consistently or bypassed
    2. **Configuration management** (dual source per D-08):
       - Grep for `get_settings()` vs `LLMConfig` vs `load_config` usage
       - Identify which modules read from .env vs YAML
       - Document the split (already in CONCERNS.md, verify current state)
    3. **Logging strategy consistency**:
       - Grep for `logging.getLogger` vs `StructuredLogger` vs `RunLogger` usage
       - Count how many files use each pattern
       - Identify files using `print()` instead of logger (already found in Phase 126 for main.py, ref)

    ### PERF-01 (async anti-patterns, per D-05) -- scan for:
    1. Sync I/O in async functions: grep for `write_bytes`, `write_text`, `subprocess.run` in async defs
    2. Unbounded collections: grep for `.append(` in long-lived objects (event_manager already found, ref)
    3. Unawaited coroutines: grep for `async def` calls without `await`
    4. Resource leaks: grep for patterns where cleanup is missing (open files, browser sessions)
