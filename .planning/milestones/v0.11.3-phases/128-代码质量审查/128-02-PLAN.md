    ### MAINT-01: DRY/SOLID Violations
    For each P1 file, identify:
    1. **SRP violations**: Does the file handle >3 distinct concerns? List each concern with line ranges.
       - run_pipeline.py: already identified as god-module (See 125-FINDINGS.md). Focus on NEW observations about which concerns can be extracted.

    ### MAINT-02: Structural Complexity
    Using radon output from Plan 01:
    1. For each function with grade C or worse: document the function name, complexity score, and WHY it is complex
    2. Identify files exceeding 500 lines and document the primary reason for size
    3. Check nesting depth: functions with >4 levels of nesting

    ### MAINT-03: Misleading Names (per D-04)
    For each P1 file, scan for:
    1. Functions where name does not match behavior:
       - Functions named `check_*` or `validate_*` that do not actually check/validate
       - Functions named `get_*` that have side effects
       - Functions named `_internal` that are called from other modules
    2. Inconsistent abstraction levels:
       - Mix of high-level service methods and low-level implementation details in same class
       - Functions at different abstraction levels called from the same context
    3. Unclear abbreviations:
       - Single-letter variables beyond loop counters (i, j, k)
       - Cryptic variable names that require context to understand

    Per D-01: If a MAINT finding was already documented in Phase 125 (e.g., variable `l` in action_translator.py), reference it with "See 125-FINDINGS.md" rather than re-analyzing.

    ### ARCH-03: Cross-cutting Consistency (three dimensions per D-08)

    #### Dimension 1: Error Handling Strategy Consistency
    Read error_utils.py, then scan ALL backend files for error handling patterns.

    For each P1 file, document:
    1. Which error handling pattern is used: `non_blocking_execute`, `silent_execute`, raw `try/except`, bare operation (no error handling), `HTTPException`, `error_response`
    2. Inconsistencies: places where error_utils helpers should be used but aren't, or where different patterns handle the same error type differently
    3. Global exception handler bypass: any route that catches and handles errors before reaching the global handler (e.g., silent error swallowing)

    Build a summary table:
    ```
    | Pattern | Files Using It | When Used |
    |---------|---------------|-----------|
    | non_blocking_execute | {list} | optional operations |
    | raw try/except | {list} | main flow errors |
    | HTTPException | {list} | route-level errors |
    | bare (no handling) | {list} | always |
    ```

    #### Dimension 2: Configuration Management (dual source)
    Read settings.py and llm/config.py.

    Document:
    1. Which modules read from Settings (.env) vs LLMConfig (YAML)
    2. Where both are accessed in the same file
    3. Configuration values that exist in both systems with potentially different values
    4. Impact of dual source: what happens if they disagree?

    Build a summary table:
    ```
    | Config Source | Files Using It | Values Accessed |
    |--------------|---------------|-----------------|
    | settings.py (.env) | {list} | {keys} |
    | llm/config.py (YAML) | {list} | {keys} |
    | Both | {list} | {which from where} |
    ```

    #### Dimension 3: Logging Strategy Consistency
    Read logger.py and run_logger.py.

    Grep all backend Python files for:
    - `logging.getLogger(__name__)` -- standard pattern
    - `StructuredLogger` -- structured JSONL pattern
    - `RunLogger` -- per-run execution log pattern
    - `print(` -- stdout bypass (already found in main.py, ref 126-FINDINGS.md)

    Build a summary table:
    ```
    | Logger Type | Files Using It | Purpose |
    |-------------|---------------|---------|
    | logging.getLogger | {count} files | {general} |
    | StructuredLogger | {count} files | {JSONL} |
    | RunLogger | {count} files | {per-run} |
    | print() | {count} files | {bypass} |
    ```

    Identify inconsistencies: files that should use StructuredLogger but use getLogger, files that mix logger types, log level misuse.

    ### PERF-01: Async Anti-Patterns (per D-05, static analysis only)
    For each P1 file, scan for:

    1. **Sync I/O in async functions**: `write_bytes`, `write_text`, `subprocess.run`, `os.path.*` operations in functions declared `async def`

    4. **Resource leak patterns**: Browser sessions, file handles, database connections that may not be cleaned up on error paths
