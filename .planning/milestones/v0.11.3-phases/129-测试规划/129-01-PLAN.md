  <action>
Read ALL four FINDINGS files completely. Apply the filter-classify-score methodology from 129-RESEARCH.md:

**Stage 1 - Filter:** For each finding, ask: "If this bug reappeared after a refactor, would an automated test catch it?"
- KEEP findings with HIGH regression protection value: correctness bugs with specific I/O expectations, error handling gaps, state mutation leaks, race conditions, performance regressions with measurable thresholds
- EXCLUDE findings with near-zero regression risk: unused imports (ruff catches), naming issues, dead code removal, missing type annotations, documentation issues, one-time fixes
- BORDERLINE: Architecture issues (test behavior not architecture), security issues (integration test with attack payloads), configuration issues (test settings control behavior)

**Stage 2 - Classify:** For each testable finding, assign test type per D-03:
- `unit`: Pure logic, no external deps (StallDetector, schemas, excel_parser, error_utils, action_translator)
- `integration`: Requires DB, API, or mocked external services (repository CRUD, pipeline stages, SSE event flow, assertion evaluation)
- `e2e`: Requires full stack running (user workflows)
- `frontend-component`: Frontend component/hook tests (useRunStream, TaskForm, client.ts)

**Stage 3 - Score by ROI:** Severity x Regression_Risk / Implementation_Cost
- Severity from original finding (Critical=4, High=3, Medium=2, Low=1)
- Regression Risk: High for mutable state, shared instances, cross-module coupling; Low for isolated pure functions
- Implementation Cost: Low (pure functions), Medium (DB tests), High (async/browser-dependent)

Create 129-FINDINGS.md with these sections:

1. **Header** matching Phase 125-128 format (phase number, date, review scope)
2. **Methodology** section documenting the filter-classify-score pipeline
3. **Statistics** table showing: total findings per phase, testable count, not-testable count, by severity breakdown
4. **Testable Findings Summary Table** -- one row per testable finding with columns: Finding ID, Source Phase, Severity, Category, Test Type, ROI Score, Brief Description. Sort by ROI descending (per D-01 severity-driven ordering)
5. **Not Testable Findings** -- grouped list with rationale for exclusion (e.g., "fix-once, no regression risk", "dead code, test after fix")
6. **Systemic Patterns Cross-Reference** -- map CP-1~CP-5 to specific testable findings that contribute to each pattern
7. **Deferred Scenarios** -- findings that are testable but require a code fix first (e.g., PreSubmitGuard dead code, assertion_service stub)

Per D-06, use the same 4-level severity format (Critical/High/Medium/Low). Per D-07, each testable entry must include: scenario name, description, source finding reference, recommended test type, priority.

Do NOT expand individual test scenarios yet -- Plan 02 and Plan 03 will do that. This task produces the filtered, classified, ordered inventory that Plans 02/03 consume.
