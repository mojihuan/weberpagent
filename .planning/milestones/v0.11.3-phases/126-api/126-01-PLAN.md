Record ruff issues and mypy errors in the findings file under "## Tool Results" section.

Step 2: Read each of the 13 files. For each file, assess risk across these 8 categories (per D-06 and RESEARCH security check matrix):

1. **Parameter validation gaps** -- Are path/query/body parameters validated with Pydantic Field constraints (min_length, max_length, pattern)? Are there bare `str` params without constraints?
2. **Error handling omissions** -- Are try/except blocks present where needed? Does error propagation reach the global handlers or get swallowed? Are there fire-and-forget patterns?
3. **Path traversal risk** -- Does any endpoint use FileResponse with a path from the database? Is the path validated with resolve() + startswith() checks?
4. **CORS/Auth gap** -- Note that ALL routes lack authentication (already in CONCERNS.md, per D-02 just verify)
5. **Code execution surface** -- Does the route trigger subprocess.run, exec(), or asyncio.create_task? What user-controlled input flows into these?
6. **SSRF risk** -- Does any endpoint accept URLs or make outbound HTTP requests based on user input?
7. **Credential exposure** -- Are credentials (passwords, API keys, tokens) logged, embedded in responses, or stored in plaintext?
8. **SSE stream error handling** -- For streaming endpoints, does the async generator have try/except/finally? Is cleanup guaranteed?

Step 3: Classify each file into P1/P2/P3 priority per D-04:
- **P1**: Routes involving code execution, external modules, or subprocess calls (batches, run_pipeline, runs_routes, external_*)
- **P2**: CRUD routes with moderate validation concerns (tasks, runs, reports)
- **P3**: Simple/read-only routes with minimal attack surface (dashboard, __init__, response.py)
