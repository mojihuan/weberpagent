### main.py Audit Checklist:
1. **CORS middleware** (lines ~75-83): Verify `allow_origins=["*"]`, `allow_methods`, `allow_headers`. Note that `allow_credentials=True` combined with `allow_origins=["*"]` is a browser-blocked combination (browsers reject credentialed requests with wildcard origin). Check if this causes issues.
2. **Lifespan** (lines ~30-50): Check `logging.basicConfig(level=logging.DEBUG)` -- is this always DEBUG or does it read from settings? What sensitive info is logged at DEBUG?
3. **Global exception handlers** (lines ~97-149):
   - StarletteHTTPException handler: Does it return consistent format? Does it include request_id?
   - RequestValidationError handler: Does it return 400 or 422? (RESEARCH says 400, verify)
   - Generic Exception handler: Does it always include stack traces? Is there a debug/production toggle?
   - All handlers: Do they properly set Content-Type to application/json?
4. **Router registration** (lines ~85-95): Are all route modules registered? Does the registration order matter?

### runs_routes.py Audit Checklist:
For EACH endpoint in the file, document:
- **Endpoint signature**: Method, path, parameters (path params, query params, body)
- **Parameter validation**: Are all string params constrained? Are there bare `str` without Field constraints? What about `step_index: int` -- is it constrained to non-negative?
- **Error handling**: Is there a try/except? Does it catch the right exceptions? Does it set appropriate HTTP status codes?
- **Security concerns**: Path traversal, code execution surface, credential handling

Key endpoints to scrutinize:
1. `POST /api/runs` (create_run): task_id validation, how is run_id generated?
2. `GET /api/runs/{run_id}/stream` (stream_run_events): SSE async generator -- does it have try/except/finally? What happens when the client disconnects? Is the event queue cleaned up?
3. `GET /api/runs/{run_id}/screenshots/{step_index}` (get_screenshot): FileResponse using `step.screenshot_path` from database -- IS THIS VALIDATED? Compare with _validate_code_path(). This is a RESEARCH Pitfall 4.
4. `GET /api/runs/{run_id}/code` (get_run_code): Uses _validate_code_path -- good pattern. Verify it's called correctly.
5. `POST /api/runs/{run_id}/execute` (execute_run_code): subprocess.run with test_file_path -- does it call _validate_code_path before execution? This is RESEARCH Pitfall 6.
6. `GET /api/runs` (list_runs): No pagination -- is this a scaling concern? (Note in findings, Low severity)
7. `_build_login_credentials`: How are credentials handled? Do they leak into logs or error responses?

### run_pipeline.py API-Layer Audit (per D-01, API layer ONLY):
Phase 125 already reviewed business logic. Only audit:
1. **Endpoint signature and HTTP handling**: How is the pipeline triggered? What HTTP parameters does it accept?
2. **SSE event publishing**: Are all event types properly formatted? Does the None sentinel work correctly?
3. **Error propagation to HTTP responses**: When the pipeline fails, does the error reach the client or get swallowed?
4. **Parameter validation**: Are all inputs (run_id, task_id, etc.) validated before pipeline execution?

Read all listed files completely. Then perform a line-by-line audit of batches.py and all external_* routes.

### batches.py Audit Checklist:
For EACH endpoint:
1. **POST /api/batches** (create_batch): Parameter validation on task_ids (is it a list? min/max length? format constraints?), concurrency parameter (is it bounded?)
2. **POST /api/batches/{batch_id}/cancel** (cancel_batch): Is batch_id validated? What if batch doesn't exist?
3. **GET /api/batches/{batch_id}** (get_batch): Is batch_id validated?
4. **Fire-and-forget pattern**: `asyncio.create_task(service.start(run_configs))` -- is there error logging? What happens if the task fails immediately? Is there a `done_callback`?
5. **Error propagation**: Can the batch appear "created" in the response but already have failed internally?

### external_assertions.py Audit Checklist:
1. **POST /api/external-assertions/execute** (execute_assertion): What parameters does it accept? Is class_name/method_name validated? What prevents arbitrary class/method execution?
2. **GET /api/external-assertions/classes** (get_classes): Does it handle the case where the external module is not loaded?
3. **GET /api/external-assertions/methods/{class_name}** (get_methods): Is class_name sanitized?
4. **Code execution surface**: `execute_assertion_method` -- what does it actually call? Does it pass user-controlled arguments to external code? Is there any sandboxing?
5. **503 guard**: Does `require_external_available()` properly catch all partial module states?

### external_data_methods.py Audit Checklist:
1. **POST /api/external-data/execute** (execute_data_method): Same questions as external_assertions
2. **GET /api/external-data/classes** (get_data_classes): Module availability check
3. **GET /api/external-data/methods/{class_name}** (get_data_methods): class_name sanitization
4. **SSRF risk**: Does execute_data_method make outbound HTTP calls with user-provided URLs? Check api_params dict handling
5. **Error format**: What format are execution errors returned in?

### external_operations.py Audit Checklist:
1. **POST /api/external-operations/generate-code** (generate_precondition_code): What inputs? Is operation_codes validated?
2. **GET /api/external-operations/operations** (get_operations): What does it return?
3. **WEBSERP_PATH check**: How does it handle the case where WEBSERP_PATH is not configured?
4. **Code generation surface**: Does it generate code that will be exec()'d? What validation exists on the generated code?
