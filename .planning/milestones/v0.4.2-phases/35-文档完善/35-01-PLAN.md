## 文档结构

```markdown
# 断言系统使用指南

## 概述

断言系统用于验证业务数据是否符合预期。本指南将帮助您理解如何配置和使用断言功能。

## 一、完整工作流程

### 1. 前端配置断言
- 在任务编辑页面，找到"断言配置"区域
- 选择断言方法（如 sell_sale_item_list_assert）
- 配置 Headers（登录身份）
- 配置 Data（数据源）
- 配置三层参数（见下节）

### 2. 执行测试
- 保存任务后，点击"执行"按钮
- AI Agent 将自动执行测试步骤
- 断言在测试步骤完成后自动执行

### 3. 查看结果
- 在报告详情页查看断言结果
- 每个断言显示：通过/失败状态、字段级验证结果

## 二、三层参数详解

断言参数分为三层，每层有不同的用途：

| 参数层 | 用途 | 示例 |
|--------|------|------|
| api_params | API 查询条件（过滤数据） | `salesOrder: 'SA'` |
| field_params | 字段验证（期望值） | `saleTime: 'now-1m'` |
| params | 向后兼容（旧格式） | 旧版参数 |

### 第一层：API 参数 (api_params)

用于控制 API 查询条件，决定从后端获取哪些数据。

**常见参数：**
- `salesOrder`: 销售单号前缀（如 'SA'）
- `articlesStateStr`: 物品状态（如 '已销售'）

### 第二层：字段参数 (field_params)

用于指定要验证的字段和期望值。

**时间字段特殊处理：**
支持 "now" 表达式，自动转换为实际时间：

| 表达式 | 含义 |
|--------|------|
| `now` | 当前时间 |
| `now-1m` | 1 分钟前 |
| `now-5m` | 5 分钟前 |
| `now-1h` | 1 小时前 |
| `now-1d` | 1 天前 |

### 第三层：通用参数 (params)

向后兼容层，旧版参数格式。建议优先使用 api_params 和 field_params。

## 三、示例：销售出库断言

以下是一个完整的销售出库断言配置：

```typescript
const assertionConfig = {
  className: 'PcAssert',
  methodName: 'sell_sale_item_list_assert',
  headers: 'main',           // 使用主账号登录
  data: 'main',              // 使用主数据源
  api_params: {
    salesOrder: 'SA',        // 销售单号前缀
    articlesStateStr: '已销售' // 物品状态
  },
  field_params: {
    saleTime: 'now-1m'       // 销售时间：1分钟前
  }
}
```

**配置说明：**
- `className`: 断言类名，PC 端使用 'PcAssert'
- `methodName`: 断言方法名
- `headers`: 登录身份（main/idle/vice/special/platform/super/camera）
- `data`: 数据源（main/a/b/c/d）
- `api_params`: API 查询条件
- `field_params`: 字段验证参数

## 四、报告解读

断言执行后，结果存储在报告的 `api_assertion_results` 字段中。

### 结果结构

```typescript
interface AssertionResult {
  success: boolean      // 执行是否成功（无异常）
  passed: boolean       // 断言是否通过
  fields: FieldResult[] // 字段级验证结果
  error: string | null  // 错误信息
  duration: number      // 执行耗时（秒）
}

interface FieldResult {
  name: string          // 字段名
  expected: string      // 期望值
  actual: string        // 实际值
  passed: boolean       // 是否通过
}
```

### 结果解读

- `success: true, passed: true` -> 断言通过
- `success: true, passed: false` -> 断言失败（数据不符合预期）
- `success: false` -> 执行出错（如超时、参数错误）

## FAQ

### Q1: 断言失败但数据看起来正确？

**原因:** 时间参数使用了固定值而非 "now" 表达式。

**解决方案:** 使用时间预设如 `now-1m` 或 `now-5m`，让系统自动计算相对时间。

### Q2: 找不到要验证的字段？

**原因:** 字段列表来自 base_assertions_field.py，可能未包含该字段。

**解决方案:** 在前端字段选择器中搜索字段名，或联系开发人员添加字段定义。

### Q3: 断言执行超时？

**原因:** 默认超时时间为 30 秒，某些复杂查询可能超时。

**解决方案:** 检查 API 参数是否过于宽泛导致数据量过大，或联系开发人员调整超时配置。

### Q4: Headers 选择错误导致数据查询失败？

**原因:** 不同的 headers 代表不同的登录账号，权限不同。

**解决方案:**
- `main` - 主账号（最常用）
- `idle` - 闲置账号
- `vice` - 副账号
- `special` - 特殊账号
- `super` - 超级账号
- `platform` - 平台账号
- `camera` - 相机账号

### Q5: 字段参数未填写期望值？

**原因:** 选中字段但未填写期望值，断言无法执行。

**解决方案:** 在前端配置界面，选中字段后必须在输入框中填写期望值。

### Q6: "now" 显示为字符串而非时间？

**原因:** 系统应在执行时将 "now" 转换为实际时间格式。

**解决方案:** 确保使用最新版本的后端代码，检查 execute_assertion_method 函数是否正确调用 _convert_now_values。

### Q7: 如何查看断言执行的详细日志？

**解决方案:** 查看后端日志，搜索 "Executing assertion" 关键字，可以看到每个断言的执行参数和结果。

### Q8: 多个断言的执行顺序？

**说明:** 断言按配置顺序依次执行，非 fail-fast 模式（一个失败不影响其他断言执行）。
