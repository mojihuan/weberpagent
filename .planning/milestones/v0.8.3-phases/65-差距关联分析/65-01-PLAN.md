**ANALYSIS-01 分析内容（核心）：**

分析目标：确认 headless 模式的 DOM 序列化差异是否直接导致 Agent 在 ERP 表格填写时定位不准（index 偏移、元素不可见等）。

使用三层证据链框架（per D-04）逐项分析：

**层 1 — 代码推理：**
分析 browser-use DOM 序列化管线中 CDP snapshot 数据对 headless/headed 的依赖关系：
1. CDP `DOMSnapshot.captureSnapshot` 在 `--headless=new` 下使用与 headed 相同的 Blink 渲染引擎（Chromium 112+），因此静态 DOM 结构和 layout 数据（bounds、styles）理论上相同
2. 但 headless 模式无真实 compositor，paint order 值可能不同（headless 使用模拟合成器）
3. AX 树 (`Accessibility.getFullAXTree`) 在 headless 下可能对"不可见"元素有不同处理（无真实屏幕导致 is_visible 判定差异）
4. Ant Design click-to-edit 的 `<input>` 在 `<td>` 未点击时是 hidden（React state），CDP layout tree 可能不包含 hidden 元素的 bounds——这是应用级行为，与 headless/headed 无关

**层 2 — 已知行为（Phase 62-64 观察记录）：**
- Phase 62: Agent 无法定位 ERP 表格中的 input 元素（销售金额字段），需要 DOM Patch 修复
- Phase 62: E2E 验证在 headless + DOM Patch 下 26 步成功
- Phase 63: 确认 f951791 强制 headless=True，覆盖 browser-use 自动检测
- Phase 63: DOM Patch 5 patches 在 DOM 序列化层面修复问题，不依赖 headed 模式
- Phase 64: 根因确认为 headless=True 覆盖自动检测

**层 3 — 补丁效果：**
- Patch 4 (`_patch_assign_interactive_indices`) 强制为缺少 `snapshot_node` 的 ERP input 分配索引——这说明 CDP snapshot 确实缺少这些 input 的数据
- Patch 2 (`_patch_paint_order_remover`) 重置 ERP 节点的 paint order 忽略——说明 paint order 确实导致了元素被过滤
- Patch 5 (`_is_textual_td_cell`) 标记 td 文本为可交互——说明 td 本身未被识别为可交互
- 所有 patch 在 headless 模式下有效解决了定位问题

**因果判定分析维度：**
1. **index 偏移问题**：index 偏移的直接原因是 ERP 元素在序列化管线中丢失（被 paint order 过滤、被 bbox 过滤、缺少 snapshot_node），导致后续元素 index 错位。需要分析这些丢失在 headed 模式下是否仍会发生。
2. **元素不可见问题**：元素不可见是因为 `snapshot_node` 缺失或 `is_visible=False`。需要分析 Ant Design hidden input 和 CDP visibility 判定是否是 headless 特有的。

**ANALYSIS-03 分析内容：**

分析目标：评估 Section 9 click-to-edit 指导在 headed 模式下是否仍需保留或需调整。

使用三层证据链框架分析：

**层 1 — 代码推理：**
1. Section 9 描述的是 Ant Design click-to-edit 交互模式：`<td>` 显示文本 → 点击 → React state 变化 → `<input>` 出现 → 输入值 → 验证
2. 这是 React/Ant Design 应用级行为，JavaScript 执行在 headed 和 headless 下完全相同（同一个 V8 引擎）
3. click 事件分发在两种模式下通过 Playwright 的 `page.click()` 正常处理
4. React state change 和 DOM re-rendering 在两种模式下走相同代码路径
5. 因此 Section 9 描述的交互模式本身不依赖浏览器模式
6. 但需考虑：如果 headed 模式下 Agent 能更好地"看到" `<input>`（因为 AX tree 更完整），Section 9 的某些 fallback 指导（如 JS evaluate 兜底）可能不再必要

**层 2 — 已知行为：**
- Phase 62 添加 Section 9 是因为 Agent 不知道如何处理 click-to-edit 表格
- Phase 62 E2E 验证在 headless + Section 9 下 26 步成功
- 没有在无 Section 9 情况下的对照测试数据
- 也没有在 headed 模式下的对照测试数据

**层 3 — 补丁效果：**
- Section 9 本身不是"补丁"，而是交互指导
- 它的价值是教导 Agent 正确的表格填写工作流，而非绕过 headless 问题
